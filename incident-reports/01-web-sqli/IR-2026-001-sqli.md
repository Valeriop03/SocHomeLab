## Executive Summary

A SQL injection attack was detected against the DVWA web application hosted on the Windows endpoint (192.168.56.20). The attempt originated from an internal host (192.168.56.30) using an automated tool. The malicious request was identified through a custom detection rule analyzing web server logs. This was a controlled lab exercise; in a production context, a successful SQLi against this application could have exposed database contents.

## Investigation

The alert (rule 100100, level 10) flagged a web request to the DVWA SQL injection endpoint containing SQL metacharacters. Pivoting on the source IP (192.168.56.30), the request was traced in the Apache access log: GET /dvwa/vulnerabilities/sqli/?id=1' OR '1'='1-- -. Two indicators confirmed malicious intent: the URL parameter contained a classic SQL tautology (OR '1'='1) with a comment terminator (-- -), and the user-agent was curl/8.12.1, indicating an automated tool rather than interactive browsing. No other request patterns from the same source suggested broader scanning at this stage. The combination of payload and tooling confirmed a deliberate SQL injection attempt — true positive.

## Impact Assessment

The target was a deliberately vulnerable application (DVWA) at security level "low". A successful SQL injection of this type could allow an attacker to bypass query logic and extract arbitrary database records (user credentials, application data). In a real environment, this could lead to data exfiltration and, depending on database privileges, further compromise. No lateral movement or privilege escalation was observed in this scenario.

---

## Recommendations

Immediate response: block the offending source IP at the network/WAF level; review whether the request returned data (response size/status) to assess success.

Hardening/prevention: use parameterized queries / prepared statements in the application; deploy a WAF with SQLi signatures; apply least-privilege to the database account.

Detection improvement: refine rule 100100 to reduce false positives (require a quote together with a SQL keyword rather than any single marker); add correlation for repeated SQLi attempts from the same source.
