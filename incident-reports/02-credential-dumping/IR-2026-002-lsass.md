## Executive Summary

A credential-dumping attempt was detected on the Windows endpoint (192.168.56.20).
An attacker abused a legitimate Windows component (comsvcs.dll) to copy the memory
of the LSASS process, where account credentials are held, into a file. The activity
was identified through a custom detection rule analyzing process-creation telemetry.
This was a controlled lab exercise; in a production environment, a successful dump of
LSASS could expose user and administrator credentials and enable further compromise.

## Investigation

The alert (rule 100200, level 12) flagged a process-creation event whose command
line contained the comsvcs.dll + MiniDump signature. Pivoting on the event details,
the full command line was:

`rundll32.exe C:\windows\System32\comsvcs.dll, MiniDump (Get-Process lsass).id $env:TEMP\lsass-comsvcs.dmp full`

This is a well-known living-off-the-land technique: the legitimate, Microsoft-signed
comsvcs.dll exposes a MiniDump export that writes a target process's memory to disk.
Here it targeted the PID of lsass.exe and wrote the output to a .dmp file in the temp
folder. The process ran at integrity level High (elevated). A correlated alert (92213)
flagged the dropped file, corroborating the finding. The combination of the comsvcs
MiniDump call against LSASS confirmed a deliberate credential-dumping attempt — true
positive.

Note on visibility: the Sysmon configuration in use (SwiftOnSecurity) does not log
Event ID 10 (ProcessAccess), so direct LSASS access was not observable; detection
relied on the process-creation command line (Event ID 1).

## Impact Assessment

LSASS holds credential material (password hashes, Kerberos tickets, and in some
configurations plaintext secrets) for accounts active on the system. A successful
dump and subsequent offline extraction could give an attacker valid credentials,
enabling lateral movement, privilege escalation, and persistence across the
environment. In this lab the dump succeeded locally; no exfiltration or follow-on
credential use was performed. No lateral movement was observed.

---

## Recommendations

*Immediate response:* isolate the affected host; rotate credentials for any accounts
that were logged on at the time of the dump; delete the .dmp artifact and verify it
was not exfiltrated.

*Hardening / prevention:* enable LSASS protection (RunAsPPL / Credential Guard) to
block memory access to LSASS; restrict and monitor use of rundll32 and comsvcs.dll;
apply least-privilege so fewer accounts can run elevated dumping commands.

*Detection improvement:* enable Sysmon Event ID 10 (ProcessAccess) filtered on
TargetImage ending in lsass.exe, to detect LSASS access regardless of the dumping
tool used — a technique-agnostic detection that complements the command-line rule.
